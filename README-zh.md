# Project Help

一个可同时用于 Codex 和 Claude Code 的 Agent Skill，通过可追溯的证据完整分析陌生项目的内容、业务、架构、运行方式、代码、运维和具体实现，帮助接收项目的人建立可工作的全貌。

[English](README.md) | [简体中文](README-zh.md)

## 技能用途

`project-help` 可以帮助 Codex 或 Claude Code：

- 默认执行完整的项目接手分析，覆盖项目内容、业务模型、架构、运行方式、数据、接口、关键流程、开发、运维、所有权、风险和安全修改路径。
- 当用户指定一个业务域、服务、模块、功能、API、事件、任务、实体、流程、目录、变更或故障时，只对该目标做实现级深挖。
- 每次分析都绘制全局整体架构 SVG，展示参与者与访问渠道、系统边界、主要运行单元、平台能力、归属数据、外部系统和关键关系。
- 每次分析都单独绘制业务架构 SVG，展示参与者目标、价值流、业务域与能力、核心实体、策略、审批和人工环节。
- 局部分析会在全局架构中高亮目标的技术位置，并在业务架构中高亮它的能力、价值流或支撑能力位置。
- 打通项目目的、领域术语、业务能力、运行架构、代码、数据、接口、测试、运维和所有权证据。
- 梳理配置、构建、启动、测试、调试、迁移、代码生成和可观测性等开发者工作流。
- 说明一次安全修改应该从哪里开始、可能影响什么，以及如何验证。
- 产出图表丰富且不设固定上限的 SVG 报告：按需增加仓库结构、上下文、业务能力、运行时、部署、依赖、事件、数据、时序、状态、安全、开发工作流、所有权和变更影响图，也可以创建清单之外的项目专属图。
- 明确区分已确认事实、推断、未知项、冲突信息和未检查区域。

## 在 Codex 中安装

安装到个人目录后，所有项目都可以使用该技能：

```bash
mkdir -p "$HOME/.agents/skills"
git clone https://github.com/gitAric/project-help.git "$HOME/.agents/skills/project-help"
```

如果只希望在某个项目中使用，请将本仓库放到：

```text
<项目根目录>/.agents/skills/project-help
```

Codex 会自动发现新安装的技能；如果技能没有出现，请重启 Codex。更多信息参见 [Codex Skills 官方文档](https://developers.openai.com/codex/skills)。

## 在 Codex 中使用

显式调用技能：

```text
使用 $project-help 为一名后端工程师完整分析并接手当前项目。覆盖项目内容、业务、全局与运行架构、数据、接口、关键流程、开发工作流、运维、所有权、风险和安全修改路径；分别产出全局整体架构和业务架构 SVG，并在证据支持时尽可能增加有独立信息价值的 SVG，不要局限于预设图表清单。
```

也可以直接用自然语言提问，让 Codex 在合适时自动选择该技能：

```text
请说明这个项目的整体结构、主要业务能力，以及各个系统之间如何交互。
```

只熟悉指定部分：

```text
使用 $project-help 只深挖结算功能。仍需产出全局整体架构和业务架构 SVG，并在两张图中高亮结算的位置；随后按证据增加调用、状态、数据、依赖、异常、测试、运维和变更影响等详细 SVG。
```

## 人类使用步骤

1. 明确接手任务：不指定范围时默认完整分析整个项目；指定一个功能时，只对该功能进行详细深挖。还可以说明接收人的角色和实际目标；如果省略，Agent 默认面向通用项目维护者建立全项目工作级理解。
2. 在 Codex 中使用 `$project-help`，或在 Claude Code 中使用 `/project-help` 调用技能。可以复用下面的请求格式：

   ```text
   帮助一名<角色>工程师接手<整个项目 / 一个指定功能>，目标是<学习 / 修改 / 排障 / 运维 / 评审>。指定功能时，只详细分析该功能；始终产出全局整体架构和业务架构 SVG，在两张图中高亮该功能，并尽可能增加有独立信息价值的其他 SVG，不局限于预设清单。
   ```

3. 先审阅任务卡和接手概览。如果 Agent 对范围、角色、术语或目标理解有误，先纠正这些边界，再继续深层报告。
4. 审阅证据，而不只看结论。检查引用的文件，并将**已确认**、**推断**、**未知**和**冲突**视为不同置信级别；把 SVG 图当作返回代码证据的地图。
5. 把两张基线 SVG 视为视觉报告的起点。项目或功能的每个有价值维度都应继续补图，并按图表索引依次审阅；局部分析时，还要确认目标在所有相关图中都被一致高亮。
6. 单独授权运行时操作或状态变更。技能默认从只读静态检查开始；启动服务、安装依赖、执行迁移、调用生产系统或修改项目文件，都需要你的明确许可。
7. 决定结果只保留在对话中，还是沉淀为长期 onboarding 文档。除非你明确要求，否则 Agent 不应把报告或图表写入被分析的仓库。

## Agent 执行步骤

1. 加载 `SKILL.md` 中的技能说明，再读取仓库级的 `AGENTS.md`、`CLAUDE.md` 等指令，并且只加载本次任务需要的技能参考资料。
2. 将任务路由为完整项目接手或局部功能深挖，再用任务卡说明范围、深度、受众、目标、假设、排除项和版本。
3. 建立证据边界。完整接手需要覆盖每个第一方顶层区域和主要业务能力；局部深挖需要先检查足够的项目广度以正确定位目标，之后不展开无关实现。
4. 每次请求都分别重建两张基线模型：全局技术/运行架构和业务价值/能力架构。局部模式必须在两张图中一致地高亮目标，并说明技术位置和业务位置。
5. 按模式继续。完整接手覆盖项目内容、业务、运行时、数据、接口、关键路径、开发工作流、运维、所有权、风险和变更准备度；局部模式只详细追踪指定功能的入口、运行装配、业务规则、数据与状态、接口、权限、错误、重试、测试、可观测性和变更影响。
6. 建立广泛、开放的 SVG 图谱，不要停在已命名的示例上。每当一种新的关系、时序、生命周期、边界、映射或影响值得单独表达时就创建一张图；拆分拥挤图，并校验和目视检查每个文件。
7. 交叉检查文字、图表、调用链、关系方向和覆盖范围。将重要结论标记为**已确认**、**推断**、**未知**或**冲突**；不要暴露密钥，也不要隐藏未检查区域。
8. 分层交付：任务与概览、两张必备基线图、图表索引和视觉阅读路径，然后是图表支撑的全项目覆盖或局部细节。说明未知项和覆盖边界；除非用户要求持久化文件，否则结果应保持为对话内或临时产物。

## SVG 图表输出

每次分析先产出两张独立的基线 SVG：全局整体架构图和业务架构图，然后在证据支持时尽可能增加不重复的 SVG。图表类型是开放的：仓库结构、运行时、部署、依赖、事件、数据模型、血缘、时序、决策、状态、安全、开发流程、可观测性、所有权、演进、风险和变更影响都只是示例，不是限制。每张图只回答一个清晰问题；图较多时必须提供图表索引和阅读顺序。
