# Project Help

一个可同时用于 Codex 和 Claude Code 的 Agent Skill，通过可追溯的代码证据，帮助你分析陌生软件项目的架构、业务能力、功能、接口、依赖和具体实现。

[English](README.md) | [简体中文](README-zh.md)

## 技能用途

`project-help` 可以帮助 Codex 或 Claude Code：

- 根据分析范围、理解深度、新人角色和实际目标自适应地组织项目导览。
- 熟悉整个项目，或者只分析一个业务域、服务、模块、功能、API、事件、任务、实体、流程、目录、变更或故障。
- 打通项目目的、领域术语、业务能力、运行架构、代码、数据、接口、测试、运维和所有权证据。
- 梳理配置、构建、启动、测试、调试、迁移、代码生成和可观测性等开发者工作流。
- 说明一次安全修改应该从哪里开始、可能影响什么，以及如何验证。
- 产出美观、清晰、可访问的 SVG 系统上下文图、组件图、流程图、时序图、状态图、数据血缘图和部署图。
- 明确区分已确认事实、推断、未知项、冲突信息和未检查区域。

## 在 Codex 中安装

安装到个人目录后，所有项目都可以使用该技能：

```bash
mkdir -p "$HOME/.agents/skills"
git clone https://github.com/gitAric/project-help.git "$HOME/.agents/skills/project-help"
```

如果只希望在某个项目中使用，请将本仓库放到：

```text
<项目根目录>/.agents/skills/project-help
```

Codex 会自动发现新安装的技能；如果技能没有出现，请重启 Codex。更多信息参见 [Codex Skills 官方文档](https://developers.openai.com/codex/skills)。

## 在 Codex 中使用

显式调用技能：

```text
使用 $project-help 为一名后端新人快速介绍当前仓库，包括领域术语、系统地图、黄金链路、开发工作流和首批阅读路径。
```

也可以直接用自然语言提问，让 Codex 在合适时自动选择该技能：

```text
请说明这个项目的整体结构、主要业务能力，以及各个系统之间如何交互。
```

只熟悉指定部分：

```text
使用 $project-help 只熟悉结算部分，达到可以开始工作的程度。说明它的最小系统上下文、入口、数据与调用流、测试、运维和安全修改边界，并将图表产出为 SVG。
```

## 人类使用步骤

1. 明确学习任务：说明范围、期望深度、新人角色和实际目标。范围既可以是整个项目，也可以是某个业务域、服务、模块、功能、API、事件、任务、实体、流程、目录、变更或故障。如果省略了某一项，Agent 会说明自己采用的合理假设。
2. 在 Codex 中使用 `$project-help`，或在 Claude Code 中使用 `/project-help` 调用技能。可以复用下面的请求格式：

   ```text
   帮助一名<角色>新人，以<快速导览 / 工作级理解 / 实现级深挖>的深度熟悉<范围>，目标是<学习 / 修改 / 排障 / 运维 / 评审>。需要输出<期望内容>，不包含<排除项>；有助于理解的图表请产出为 SVG。
   ```

3. 先审阅任务卡和快速导览。如果 Agent 对范围、角色、术语或目标理解有误，先纠正这些边界，再要求继续深入。
4. 审阅证据，而不只看结论。检查引用的文件，并将**已确认**、**推断**、**未知**和**冲突**视为不同置信级别；把 SVG 图当作返回代码证据的地图。
5. 只在有价值时选择下一层：可以继续要求追踪黄金链路、一个局部范围、开发者工作流、安全修改地图、故障路径、接口契约、状态转换、测试路径或上手练习。
6. 单独授权运行时操作或状态变更。技能默认从只读静态检查开始；启动服务、安装依赖、执行迁移、调用生产系统或修改项目文件，都需要你的明确许可。
7. 决定结果只保留在对话中，还是沉淀为长期 onboarding 文档。除非你明确要求，否则 Agent 不应把报告或图表写入被分析的仓库。

## Agent 执行步骤

1. 加载 `SKILL.md` 中的技能说明，再读取仓库级的 `AGENTS.md`、`CLAUDE.md` 等指令，并且只加载本次任务需要的技能参考资料。
2. 按范围、深度、受众和目标路由任务。给出简洁的任务卡，写明假设、排除项、相邻上下文，以及时间或文件预算。
3. 建立证据边界：识别仓库根目录和版本、生成或第三方代码、权威文档、清单文件、有效注册点、数据结构、测试和证据缺口。默认先做只读静态检查。
4. 先建立新人友好的心智模型，再进入实现细节：说明项目目的、参与者、可观察结果、核心术语、有意义的组件边界、一条代表性黄金链路，以及首批阅读或运行路径。
5. 只追踪到请求所需的深度。按需连接运行时装配、业务规则、数据和状态、接口、权限、错误、重试、测试、可观测性、所有权证据和变更影响；必须确认模块被有效装配后，才能把它描述为当前行为。
6. 只创建能明显提升理解的最小图表集合。产出独立、自包含的 SVG 文件，遵循 `references/diagram-patterns.md`，使用 `scripts/validate_svg.py` 校验每个文件，并目视检查清晰度、间距、对比度和方向是否准确。
7. 交叉检查文字、图表、调用链、关系方向和覆盖范围。将重要结论标记为**已确认**、**推断**、**未知**或**冲突**；不要暴露密钥，也不要隐藏未检查区域。
8. 分层交付：先给出答案或快速导览，只在需要时追加更深证据，说明剩余未知项和成本最低的下一步学习动作，并在达到请求深度后停止。除非用户要求持久化文件，否则结果应保持为对话内或临时产物。

## SVG 图表输出

当图表能明显帮助理解时，该技能会产出独立 SVG 文件，而不是 Mermaid 源码。内置渲染器为流程图和时序图提供统一视觉风格，校验器会在交付前检查可移植性、可访问性元数据、内部引用和基本可读性。
